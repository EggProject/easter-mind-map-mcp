Easter Mind Map MCP — headless, párhuzamos mind‑map tervezés

Egy TypeScript/Bun alapú MCP szerver, amely kizárólagos integrációs rétegként a MindGeniusAI HTTP/SSE backendje elé épül. Ez a report a fejlesztő agent‑csapatnak szóló terv vizuális összefoglalója; a teljes, átadható terv‑prompt a kísérő markdown fájl.

Bun + TypeScript 6 MCP 2025‑11‑25 (RC‑forward 2026‑07‑28) MCP TS SDK v1.x Prettier · ESLint · Lefthook Valós kódból verifikálva

Mit építünk

Stateful MCP adapter + job‑orchestrátor egy stateless generáló engine fölé: tartós planningId, queue, concurrency, SSE‑reducer, export.

Miért MCP

Az LLM ne hívja közvetlenül a backendet. Minden művelet MCP toolon/resource‑on át — a tool‑szövegezés maga is prompt.

Hogyan

10 specializált agent, 6 fázis, explicit kapuk, web‑verifikált stack és független teljesség‑ellenőrzés.

A hozzáadott érték a stateless engine fölé

MCP tool interfész
+ explicit, tartós planningId / runId / documentId
+ perzisztens állapot (terv, run, esemény, dokumentum)
+ job queue + concurrency limit (global / owner / plan)
+ MindGeniusAI SSE feldolgozás + mind‑map state reducer
+ dokumentum‑koordináció (upload + RAG index)
+ cancellation + retry + idempotencia
+ OPML / PNG export + AI‑olvasható flow‑leírás
+ biztonság (ownership, OAuth, SSRF, quota)
+ upstream felügyelet (health‑check + auto‑start)
A forrás‑mind map két különálló scope‑ra bomlik (1) az új MCP adapter szerver (Bun/TS6, a mi kódunk), és (2) az ./original‑MindGeniusAI upstream repón végzett módosítások (MiniMax provider + Claude Code CLI tmux‑szal). A kettő külön kódbázis — nem keverjük.

Upstream kontraktus — a MindGeniusAI valós állapota

Ezt nem a spec‑tervezetből vettük, hanem a valós forráskódból fejtettük vissza. A Reverse‑Engineering agent első dolga visszaigazolni a friss main‑ből.

RAG / dokumentum folyamat

In‑process memória index (Map<fileName, chunks>), nem perzisztens, instance‑hez kötött.

# 1) feltöltés (multipart, csak PDF, ≤10MB)
POST /api/uploadFile      → { fileName }

Idempotens a fileName‑re; LRU eviction 20 felett; nincs cross‑name dedup.

MCP toolok — a „generáló endpointok"

A szövegezés maga is prompt Minden tool‑név, leírás és paraméter‑leírás úgy van megírva, hogy az LLM külön külső promptolás nélkül hívja láncba. Minden leírás megmondja: mit csinál, mikor hívd, mit ne, és mi a nextAction. Ezt a 16. fejezet eval‑ja validálja (fázis‑kapu).

mindmap_continueMeglévő terv folytatásaBetölti a korábbi messages + mindMap‑et; plan‑szinten soros.

A flow‑recept (mindmap_guide)

1) mindmap_create(prompt[, documentId])
   → jegyezd meg a planningId-t
2) ismételd: mindmap_get_status(planningId)
   amíg status ∈ {completed, failed}
3) finomítás: mindmap_continue(planningId, instruction)
   → vissza a 2)-re
4) mindmap_get_result(planningId, 'outline'|'markdown')
5) KÖTELEZŐ ZÁRÁS:
   mindmap_export(planningId, ['opml','png'])

07 · Megbízhatóság

Perzisztencia, queue & konkurrencia

A stateless engine fölé tartós állapot, soros plan‑módosítás és optimista verziózás kerül. A „jó ötletek + modern" döntés szerint Redis‑mentes alternatívát ajánlunk.

Tárolás (ajánlás + alternatíva)

Durable
PostgreSQL (plan/run/event/idempotency/ownership)
Queue
pg‑boss (Postgres, Redis‑mentes) vagy BullMQ (Redis)
Dokumentum
objektumtároló / kontrollált FS
Memória
csak AbortController + aktív worker ref

A queue‑döntés ADR‑ben rögzítendő. Bun 1.3 natív Postgres kliense kihasználható.

Concurrency szabályok

  • különböző planningId → párhuzamos
  • azonos planningId módosítói → sorban (perPlanning = 1)
  • dokumentum‑index → külön queue
  • backpressure; 429/5xx → exp. backoff
  • config nélküli korlátlan párhuzamosság tilos

Konzisztencia, idempotencia, retry

// optimista zár run-induláskor
baseVersion = plan.version
workingSnapshot = clone(plan)
// commit CSAK ha:
if (plan.version === baseVersion) commit()
else conflict() // nem írja felül az újabbat
Biztonságos retry Csak: kapcsolatfelépítési hiba, timeout az első érdemi esemény előtt, 429, átmeneti 502/503/504. Nem retry‑olunk olyan futást, amely már dolgozott fel mind‑map eseményt. Partial event sosem kerül a commitolt állapotba.

SSE no‑leak — a mind map explicit követelménye

„Ne maradjon sok SSE kapcsolat" Runonként egyetlen upstream SSE kapcsolat, AbortController‑hez kötve; garantált lezárás done/failed/cancel/timeout/shutdown esetén; idle‑ és max‑életkor timeout; a concurrency limit megakadályozza a felhalmozódást. A QA‑nak explicit leak‑auditja van: terhelés alatt a nyitott socketek száma stabil, futások után 0.

08 · Export & biztonság

Export, flow‑endpoint és biztonság

OPML + PNG export

OPML

Determinisztikusan generálható az outline‑ból (<outline text="..."> fa). Tiszta függvény a packages/shared‑ben, unit‑tesztelve.

PNG — ADR‑döntés

Az upstream export kliens‑oldali (AntV X6), ezért headless megoldás kell:

(A) Headless render X6‑tal (pixelpontos, böngészőt igényel)

(B) Szerver‑oldali layout (ELK/d3‑flextree) → SVG → resvg/sharp ajánlott

(B) a headless tisztaságért; (A) fidelity‑fallback.

Flow‑endpoint (mindmap_guide) A dokumentált upstream folyamat + a megtervezett MCP toolok AI‑olvasható receptje, amely kötelezően OPML+PNG exporttal zár. A mind map 0f8287b7 követelménye.

09 · Külön scope

Upstream módosítások — ./original-MindGeniusAI

A valós upstream kódbázison (Hono + Node + AI SDK v5 + pnpm), nem az MCP szerveren. A gyári viselkedés nem törhet.

MiniMax provider config‑bound

A gyári providerek (openai/anthropic/deepseek/moonshot) változatlanok. Új, configból kapcsolható provider:

// MiniMax OpenAI-kompatibilis → Moonshot-minta
ProviderName += 'minimax'
createOpenAI({ baseURL, apiKey }).chat(model)

// env
MINIMAX_API_KEY
MINIMAX_BASE_URL = https://api.minimax.io/v1
MINIMAX_MODEL    = MiniMax-M2 | M2.x | M3  // verifikáld
// per-request: X-LLM-Provider: minimax

Embedding hiányában a RAG szépen degradál (meglévő logika).

Claude Code CLI tmux‑only

Interaktív claude session programozott vezérlése — kemény megkötésekkel:

  • CSAK tmux: new-session / send-keys / capture-pane
  • TILOS az Agent SDK
  • TILOS a claude -p (print/headless)

Determinisztikus session‑életciklus, timeout/hibakezelés, strukturált pane‑kinyerés. Statikus teszt igazolja: nincs Agent SDK import, nincs claude -p.

11 · Definition of Done

Elfogadási kritériumok

A spec‑tervezet 15 magpontja (frissítve) + 8 új kritérium a mind map kiegészítéseiből. F5 zárásához mind zöld.

MCP mag

  • Kizárólag MCP toolokon át használja az LLM a backendet
  • mindmap_create azonnal tartós planningId
  • Több terv párhuzamosan
  • Azonos terv két módosítása nem párhuzamos
  • Plan‑ek állapota izolált
  • Restart után a planek visszatölthetők
  • mindmap_continue messages + mindMap alapján
  • Minden SSE eseménytípus kezelve
  • Partial/hibás run nem ír felül commitolt verziót
  • mindmap_cancel abortálja az upstream streamet
  • Global/owner/plan concurrency limit
  • Dokumentum párhuzamos indexelése = egy embedding‑futás
  • Nagy eredmény resource‑ként
  • Ownership minden planhez/dokumentumhoz
  • MCP Tasks nélkül is teljesen működik
